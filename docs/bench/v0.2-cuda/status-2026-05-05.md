## Phase 6 — INT4 desc_act g_idx 修复（commit `dad43e3`）

**根因**：M2 / Llama-3.1-8B-GPTQ-INT4-Marlin 是 `desc_act=true`
（act-order）格式，但 ferrum 的 Marlin/Triton 路径没读 `g_idx`，
导致用错 scale/zero。

**修复**：`crates/ferrum-quantization/src/native_safetensors.rs` 检测
`desc_act=true` 时，**load 时 CPU dequant 到 f32 + 放回 original-K
顺序**，构建 `DenseLinear<B>` 走 cuBLAS f16 GEMM。验证数学：vLLM
exllama 的 `argsort + gptq_shuffle` 化简为 `y[n] = Σⱼ x[j]·dequant(
qweight[j,n], scales[g_idx[j],n])` — 即 disk_k 直接是 original_k，
无需行置换。

**结果（M2 INT4 内容正确）：**
- c=1: 50.2 tok/s（持平 M1 FP16）
- c=4: 151.5
- c=16: 350.9

VRAM: ~16 GB（FP16 dense weights）

## Phase 7 — 真实 batch scaling 诊断

`FERRUM_BATCH_DECODE_PROF=1` 测出：
- c=4 iter median = 21.98 ms
- c=16 iter median = 29.03 ms
- Δ / 12 seqs = **0.59 ms/seq**（kernel-level scaling 良好）

vs vLLM iter ~6.84/8.65 ms — **ferrum 每 iter 比 vLLM 慢 3.2-3.4×**，
不是 scaling 问题，是 per-iter 速度问题。

## Phase 8 — perm-aware Marlin INT4（commit `48bf17b`）

恢复 INT4 GEMM 速度。新流程：
- LOAD：检测 `desc_act` → `perm = argsort(g_idx)` → 解包 INT4 →
  按 perm 重排行 → 重新打包 INT4 → 标准 Marlin repack → store perm
- RUNTIME：`marlin_gemm_with_perm` 分配 fp16 scratch → 启动
  `gather_columns_f16`（`output[m,j] = input[m, perm[j]]`）→ 标准
  `marlin_gemm` on gathered input

数学等价：`y[n] = Σⱼ x[m,perm[j]]·dequant(qweight_sorted[j,n]) =
Σ_k x[m,k]·dequant(qweight[k,n])` ✓ 与未置换 GEMM 完全相同。

**c=1 verify**：M2 复现 "A physicist walks into a bar..."（与 dequant→
Dense 字节相同）。c=4 16/16 correct，c=16 64/64 correct。

**真实 Phase 8 数字（中位数 3 reps）：**

## Phase 9 — multi-slot CUDA graph cache（commit `fb21ebd`）

之前 graph 模式被诊断为 "multi-replay SIGSEGV"，但实测发现 **是 cuBLAS
GEMM 在 captured graph 里的状态问题** — Phase 8 把 GEMM 路径切到
Marlin，意外修了 graph 稳定性。

进一步发现 c=16 graph 模式 4× 回归 = single global slot 在 m_padded 切换
（1/2/4/8/16）时被频繁 capture/replace。

**原生 CUDA 验证**（`scripts/graph_upload_bench.cu`）：
- 320 launches × 500 iter 交替两个 graph，~0.26 ms/iter
- cuGraphUpload-per-replay 影响 ≈ 0（-0.014/+0.002 ms 噪声内）
- **多 slot 安全**

**重构**：`static DECODE_GRAPH: Option<>` → `HashMap<u64, GraphSlotRaw>`，
Backend trait 添加 `key: u64` 参数。单条用 `key=0`，batched 用
`m_padded as u64`。

**实测**：fresh server c=16 graph = 479.4 tok/s（持平 no-graph 484）。
graph 工作正常但 0% 提升。

### 为什么 graph 没用？（已证伪 launch overhead 假设）

CUDA stream 是 FIFO 异步，launch overhead 被 kernel 执行掩盖。
- ferrum 每 iter 320 launches × 5us host dispatch = 1.6ms 总
- 每 kernel 平均 GPU 时间 = 17ms / 320 = **53us**
- GPU work (53us) >> dispatch (5us)，GPU 不空闲

**Graph 在 ferrum 当前工作负载上是结构性死路。** 只在小模型（每 kernel
< 5us）或 spec decode draft 等场景才有意义。

## Phase 10 — Marlin tile 调优实验（commits `38ed09f`, Task #17）

**假设**：Marlin tile (64x256) 比 (128x128) 在 m=16 快 2× 基于 5%
带宽利用率推算。

**实测**：
- default (128x128): c=4=257.8, c=16=483.3
- 64x256: c=4=255.6, c=16=482.1
- Δ = -0.2% / -0.9% — 完全噪声内

**结论**：**Marlin INT4 是 compute-bound 不是 memory-bound**。INT4→FP16
dequant + tensor-core mma 算术强度高，tile shape 改 IO 帮助有限。带宽
利用率推算错了，没考虑 dequant 算术。

`FERRUM_MARLIN_TILE` env 保留供未来 A/B 测试。

## Phase 11 — c=1 op-profile 实测（FERRUM_DECODE_OP_PROFILE）

m=1 单 token decode 内核分布：

```
[layer-profile] 32 layers total=9 ms avg=287 us min=286 us max=297 us
[op-profile] matmuls (Linear::forward):  128 calls 6 ms (avg 52 us)   ← 73%
[op-profile] qk_norm_rope:                32 calls   (avg 21 us)        7.4%
[op-profile] norms:                       64 calls   (avg  8 us)        5.6%
[op-profile] flash_attn:                  32 calls   (avg 14 us)        5%
[op-profile] other:                       96 calls   (avg  4 us)        4%
```

**Marlin GEMM = 73% iter 时间**。任何想 2× iter 速度的优化都必须改
Marlin。

跟 vLLM 对比（同 RTX 4090 同模型）：
- ferrum Marlin call avg @m=1: **52 us**
- vLLM 估算 @m=1: ~39 us（基于 1.34× 总 iter 比）
- ferrum @m=16: **~98 us**
- vLLM @m=16: ~49 us（基于 2× 总 iter 比）

**vLLM Marlin kernel 内部比 ferrum 快 2×**（在 m=16 上）。
