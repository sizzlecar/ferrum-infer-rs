# vLLM gptq_marlin port — Phase 12 conclusion (2026-05-05)

**Branch:** `bench/v0.2-cuda` HEAD `06b9d75`
**Pod:** Vast.ai RTX 4090 (sm_89), Llama-3.1-8B GPTQ-INT4 (M2)
**Bench:** `bench/v0.2-cuda/bench_clean_c16.sh` 3 reps, c=16, 32 prompts × 128 tokens

## TL;DR

Ported vLLM's `gptq_marlin.cu` + 19 kernel template instantiations
generated by `generate_kernels.py 8.9`. Wired into ferrum-kernels as
`extern "C" ferrum_marlin_mm_f16_u4b8`, gated on `FERRUM_VLLM_MARLIN=1`.
Verified the kernel runs (one-time launch print) and produces correct
output (sample request: "In a world of wires and circuits, a lone robot
named Zeta..."). **But throughput is identical to ferrum's existing
IST-DASLab Marlin** — within 1% noise across three perf-knob
configurations.

The original Phase 11 estimate ("vLLM Marlin call avg @m=16: 49us vs
ferrum 98us") is **not reproducible** in actual end-to-end bench. Either
the 49us came from a different workload (different N/K/group, different
arch) or the kernel-isolated benchmark didn't translate to real decode
iter time.

## Numbers (median of 3 reps, warmup-excluded)

| Configuration | tok/s | TPOT_p50 (ms) |
|---|---|---|
| Baseline (ferrum IST-DASLab Marlin) | **520** | 22.7 |
| vLLM Marlin (default) | **517** | 22.8 |
| vLLM Marlin + use_atomic_add | **517** | 22.6 |
| vLLM Marlin + use_fp32_reduce | panic (needs C_tmp buffer) | — |

Within run-to-run noise (±5%).

## What was done

1. **Generated 19 kernel template instantiations** via vLLM's own
   `generate_kernels.py 8.9` (sm_89 Ada Lovelace = RTX 4090). Covers
   full dtype matrix (FP16/BF16/S8/FE4M3 × U4/U4B8/U8B128/FE2M1/FE4M3
   × FP16/BF16). Only `sm80_kernel_float16_u4b8_float16.cu` is reachable
   at runtime for our M2 model (FP16 act + GPTQ-INT4 weight + FP16
   output), but ALL had to be compiled because `kernel_selector.h` in
   `marlin.cu` references every variant in if/else chain — compiler
   refuses to drop unused.

2. **Stripped torch::Tensor entry points** from `marlin.cu` and
   `gptq_marlin_repack.cu`. Wrapped them in `#if false ... #endif` for
   reference. Removed `#include "core/registration.h"` and
   `TORCH_LIBRARY_IMPL_EXPAND` macros. Added a thin extern "C" wrapper:

   ```c
   extern "C" void ferrum_marlin_mm_f16_u4b8(
       const void* A, const void* B, void* C, ...,
       int prob_m, int prob_n, int prob_k, int lda,
       void* workspace, bool has_act_order, bool is_k_full,
       int num_groups, int group_size, int dev, cudaStream_t stream,
       int sms, bool use_atomic_add, bool use_fp32_reduce);
   ```

3. **Build infrastructure** in `crates/ferrum-kernels/build.rs`:
   - New `compile_vllm_marlin()` modeled on existing `compile_marlin()`
   - Compiles 22 .cu files (3 + 19 instantiations) via nvcc to .o
   - Bundles into `libvllm_marlin.a` (final size 94 MB; full kernel
     set is heavy)
   - Total build time: 18m23s on first build (incremental rebuild < 1m)
   - Gated on `vllm-marlin` cargo feature

4. **Rust FFI binding** at `src/vllm_marlin.rs`. `extern "C"` decl plus
   safe-ish `launch_marlin_mm_f16_u4b8` wrapper.

5. **Production integration** at `cuda.rs::marlin_gemm_with_perm` —
   reads `FERRUM_VLLM_MARLIN` env at runtime, dispatches to vLLM port
   instead of ferrum's IST-DASLab kernel when set. Passes
   `has_act_order=false` because ferrum's perm-aware load already
   pre-permuted weights + we gather A by perm before the GEMM.

6. **Standalone correctness check**: started server with `FERRUM_VLLM_MARLIN=1`,
   sent a single coherent prompt, confirmed:
   - `[vllm-marlin] launch #0 m=37 n=6144 k=4096 group_size=128`
     log line — kernel was actually invoked
   - Output: "In a world of wires and circuits, a lone robot named
     Zeta stumbled upon a hidden garden..." — coherent, no garbage

## Why the port produced no speedup

The exec-batch-decode-prof from prior engine work showed:
- model_call (= GPU forward + final sync + to_vec) = **14.3 ms**
- Of that, ferrum's Marlin GEMM consumes ~65% = **9.3 ms** (per c=16
  op-profile, status-2026-05-05-c16-opprofile.md)

If vLLM Marlin were truly 2× faster, model_call should drop to
14.3 - 9.3/2 = **9.7 ms** → engine iter ~11.4 ms → throughput ~1400
tok/s. We got 517 (vs baseline 520).

This means **either**:
- (a) vLLM Marlin is NOT 2× faster on RTX 4090 + this exact M/N/K/group
  combination (Llama-3.1-8B qkv proj 4096×6144, gate_up 4096×28672, etc.)
- (b) ferrum's IST-DASLab Marlin is already at the kernel-craft ceiling
  for these shapes (consistent with prior Phase 10 finding that "Marlin
  INT4 is compute-bound — tile tuning gave 0% gain")

Either way, the "vLLM Marlin = 2× speedup" working assumption is
falsified for this workload.

## What's next (post-Phase-12)

Phase 12 closes as **correct port, zero perf win**. Code is kept (gated
behind `vllm-marlin` feature) for future rebench on different shapes
(Qwen2.5, larger MoE, SM_90 Hopper).

Remaining levers ranked by ROI for closing the gap to vLLM 1597 tok/s
(currently at 520 = 33%):

1. **Chunked prefill** — engine architecture: pack multiple prompts'
   prefill chunks alongside decode in one forward, never block decode.
   ~1-2 weeks. Helps TTFT mostly, mild throughput improvement (~10-15%).

2. **FULL CUDA Graph for decode** — vLLM uses `cudaGraphLaunch` of an
   entire decode iter, claimed 3× speedup at small m. Ferrum has
   piecewise CUDA Graphs (16 graphs covering pre_attn / post_attn etc.)
   but tried + abandoned full-iter graph multiple times due to SIGSEGV
   in `cuGraphLaunch` (per `project_v02_cuda_bench.md`). Would need
   a deep dive.

3. **Persistent cuBLAS workspace pool** — minor, probably <5%.

4. **Per-tile profile-driven dispatch** — vLLM's marlin tries multiple
   tile configs at runtime and picks the fastest. Ferrum hardcodes
   one. Maybe adding this to ferrum's IST-DASLab port gives 5-15%.
