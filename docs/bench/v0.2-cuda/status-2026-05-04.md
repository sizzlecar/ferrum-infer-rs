# v0.2 CUDA Bench — 状态快照 2026-05-04

## 目标

ferrum-infer-rs 对标 vLLM，在 NVIDIA RTX 4090 上的 LLM serving 性能。
用户目标：在相同硬件 + 模型 + 工作负载下超越 vLLM。

主要测试模型：

| Tag | 模型 | Precision | 体积 |
|---|---|---|---|
| M2 | Llama-3.1-8B-Instruct-GPTQ-INT4-Marlin | INT4 | ~5.7 GB |
| M1 | Llama-3.1-8B-Instruct | FP16 | ~16 GB |
| M3 | Qwen3-30B-A3B-MoE-Instruct | FP16/INT8 | （CUDA boot 失败，未列入主跑） |

## 测试方法

**硬件**：vast.ai 租用 pod，RTX 4090，CUDA 12.4。
（具体 pod ID 和 SSH 端点见本地 `.env.local`，未提交。）

**Bench harness**：vendored `vllm bench serve`（vLLM 0.20+ 的 CLI 子命令）。

**调用链**：
```
run_sweep.sh（一次性启动 server，扫 c × repeat）
  → run_cell.sh <engine> <model_tag> <c> <repeat> <port>
    → vllm bench serve --backend openai-chat --base-url http://127.0.0.1:$PORT
       --endpoint /v1/chat/completions --num-prompts 4*c --max-concurrency $c
       --request-rate inf --temperature 0 --top-p 1
```

**Server 启动参数**：
```bash
CUDA_VISIBLE_DEVICES=0 \
FERRUM_KV_CAPACITY=2048 \
FERRUM_MAX_BATCH=32 \
ferrum serve --model <model_dir> --port 8800
```

**Workload 矩阵**：
- 并发 c ∈ {1, 4, 16, 32}
- 每 c 重复 3 次（取中位数）
- prompts 数量 = c × 4，封顶 128
- c=1: prompt 128 tok / output 512 tok
- c≥4: prompt 512 tok / output 256 tok
- 输入 prompts：`bench/v0.2-cuda/prompts.jsonl`（ShareGPT 子集，token 上限 ~579）
- 同 pod 跑 vLLM 0.20 作 baseline，跑相同 workload

**测量指标**：
- output_throughput (tok/s) — 整体输出 token 吞吐
- median_tpot_ms / p99_tpot_ms — Time Per Output Token 中位数 / P99
- median_ttft_ms / p99_ttft_ms — Time To First Token

**FERRUM_KV_CAPACITY=2048** 的原因：默认 512 不够 prefill 512 + decode 256，
会触发 "KV cache overflow on layer 0: would write tokens [0..579) but capacity is 512"
panic。2048 留出 prompt(≤512) + output(≤512) + 余量。

**FERRUM_MAX_BATCH=32**：覆盖 c=32 上限。c=4 实际跑 m=4。

## 核心流程

请求生命周期（OpenAI chat completions 路径）：

```
HTTP /v1/chat/completions  (ferrum-server::axum_server)
  └→ tokenizer.encode → InferenceRequest

ContinuousBatchEngine::run_iteration  (ferrum-engine)
  └→ tokio::Mutex iteration_lock  (整个 iter 串行)
    └→ ContinuousBatchScheduler.next_batch(BatchHint)
       └→ 返回 BatchPlan { prefill_ids, decode_ids }
    └→ process_batch(batch)
        ├→ prefill: 每个 ID 单独跑 (prefill_internal)
        └→ decode: run_batch_decode(decode_ids)
            ├→ 构造 DecodeInput[]（kv_cache_handle + last_token）
            ├→ executor.batch_decode → model.decode_batch_internal(batch)
            │    └→ LlamaFamilyModel::decode_batch_internal
            │        ├→ write_u32 → batch_positions, batch_kv_lens_pre, batch_kv_lens_post
            │        ├→ embedding_lookup → residual
            │        └→ for li in 0..32:
            │            forward_layer_batched_decode (rms → qkv linear → qk_norm_rope_batched
            │             → kv_cache_append_batched (K, V) → flash_attention_batched
            │             → o_proj → fused_add_rms_norm → gate_up linear
            │             → fused_silu_mul → down_proj → residual_add)
            │        └→ final rms_norm → lm_head → batch_logits → host Vec<Vec<f32>>
            └→ for each (rid, decode_output):
                ├→ logits.to_vec_f32  (host copy)
                ├→ sequences.write() lock
                │   └→ sample_with_processors → next_token
                ├→ scheduler.update_decode_progress
                ├→ send_stream_update.await  (mpsc → axum SSE)
                └→ should_stop check → complete_request
```

后端实现走 `Backend<B>` trait → `CudaBackend`（ferrum-kernels）。
INT4 矩阵乘走 Marlin（IST-DASLab fork），auto-tile 选择：
- prob_m ≤ 16: thread_k=128, thread_n=128
- prob_m > 16: thread_k=64, thread_n=256

`run_iteration` 是 c=4 时所有 4 个并发请求共享的；每 iter 产出 m=4 个 token
（每请求 1 个）。

### Phase 4d（CUDA Graph capture wiring）— 已搁置

代码在 commit `96eae06` + 后续，env-gated `FERRUM_BATCHED_GRAPH=1`，默认关。

#### 已验证的发现

**1. 4 个 cudarc graph 基线测试（pod，`/workspace/graph_tests.log`）**：

| 测试 | 结果 | 说明 |
|---|---|---|
| `cudarc_graph_default_pool` | FAIL | `CUDA_ERROR_STREAM_CAPTURE_ISOLATION` — cudarc 已知要求，必须先 `disable_event_tracking()` |
| `cudarc_graph_no_event_tracking` | OK | event_tracking off + 6 次 replay 全通过 |
| `cudarc_graph_with_many_allocs` | OK | 200 个 weight bufs + 6 replays 通过 |
| `cudarc_graph_like_ferrum` | OK | cuBLAS device pointer mode + 多 PTX module + htod-before-replay 通过 |

→ 基础 graph 基础设施 OK；ferrum 已经在 `replay_last_graph` 调用 `disable_event_tracking()`。

**2. 共享 host array bug（cudarc test，pod 上跑通过）**：

新增测试 `cudarc_graph_shared_host_array_multi_memcpy`（commit 58872d5）：
两次 captured `stream.memcpy_htod` 都从同一个 `Box<[u64; 4]>` 读，中间用不同
内容覆盖 host array 后 launch graph。期望 dev1=[1,2,3,4]、dev2=[5,6,7,8]，
实际：

```
[shared-host-multi] dev1 = [5, 6, 7, 8]
[shared-host-multi] dev2 = [5, 6, 7, 8]
BUG CONFIRMED: both dev buffers read latest host value (race)
```

→ 实锤 graph 记录的是 host 指针，不是 capture 时的内容。
ferrum 32 层 batched_decode 共用 `batched_host_cache_ptrs` → replay 时所有
captured memcpy 都读到 layer 31 的指针 → kernel 读到错的 cache 指针 →
MISALIGNED_ADDRESS。

**3. Slot fix（commit 9b79001）— 已修复 host-array 共享问题**：

每个 call site 用独立的 slot 区间：
- K-append: `slot = li`
- V-append: `slot = li + MAX_LAYERS_FOR_GRAPH`
- flash_attn: `slot = li`（用独立的 k_ptrs/v_ptrs 数组）

host array 从 `Box<[u64; 64]>` 扩到 `Box<[u64; 2 * MAX_LAYERS_FOR_GRAPH * 64]>`
(8192 个 u64 = 64KB)。

效果：第 1 次 post-capture replay 通过 `CUDA_SUCCESS`，但 c=4 bench 0/16 ok，
第 2 次 replay 时 `cuGraphLaunch` 之后进程 silent SIGSEGV。

**4. sync-after-launch 诊断（commit 828c175）**：

在 `replay_last_graph` 的 `cuGraphLaunch` 之后立即调 `cuStreamSynchronize`。
跑出来：

```
post-capture replay (iter 4):
  [replay] cuGraphLaunch returned: CUDA_SUCCESS
  [replay] post-launch cuStreamSynchronize: CUDA_SUCCESS  ← OK

iter 5 (1st pure replay):
  [replay] cuGraphLaunch returned: CUDA_SUCCESS
  [replay] post-launch cuStreamSynchronize: CUDA_ERROR_ILLEGAL_ADDRESS  ← KERNEL 崩了
```

→ 第 2 次 replay 的 launch 提交成功，但 kernel 执行时读越界。compute-sanitizer
报 "host-side error"（不是 kernel-side memory error），暗示 captured memcpy 自
身的问题。

**5. Pre-populate fix（commit af980d9）— 把 captured memcpy 全部移出 graph**：

新 trait method `populate_batched_pointers` 在 forward 之前一次性 sync memcpy
（cuMemcpyHtoD_v2，null stream，不被 capture）填 device scratch。captured
graph 只剩 kernel launches。

效果：第 2 次 replay 仍然 hang/SIGSEGV（cuGraphLaunch 不返回）。

**6. cuGraphUpload-per-launch（commit 0b7c762）+ AUTO_FREE_ON_LAUNCH**：

每次 launch 前调 cuGraphUpload 重新 upload。

效果：14 次连续 cuGraphLaunch 全 `CUDA_SUCCESS`，第 15 次 SIGSEGV。
`grep -c "replay ok"` = 14。

**7. flags=0（commit bff86e9，后由 08b0b46 revert）— 试 AUTO_FREE_ON_LAUNCH 影响**：

效果：只撑过 4 次 launch 就 SIGSEGV（vs AUTO_FREE 的 14 次）。
→ AUTO_FREE 帮一些忙，flags=0 反而更差。

**8. populate-cache（commit 4732108，后由 08b0b46 revert）— 试图缓存 populate**：

逻辑：`batch_pointers_for: Option<Vec<String>>`，只在 batch composition 变化时
重新 populate。

效果 c=4 OFF: **176.9 tok/s**（vs revert 后的 221.9），TPOT 18.0ms。

→ Bug：`new_context()` 每次 decode_batch_internal 都新建一个 CudaState，
device scratch 是 None。我的缓存逻辑跳过 populate，但新 ctx 的 scratch 还是
None → kernel 检查失败 → 走 fallback 的 per-item 路径 → 比 batched 还慢。

**9. revert（commit 08b0b46）— 恢复 captured-memcpy 版本**：

放弃 populate fix，回到 `stream.memcpy_htod`（async，被 capture），保留 slot
fix。FERRUM_BATCHED_GRAPH 默认关，graph 路径不再触发。

c=4 OFF: 221.9 tok/s（vs Phase 4 era 235）—— slot fix 引入约 5% TPOT 开销
（host array 8192 entries vs 64，+ slice/slice_mut view 调用，+ 每个 call
site 的 slot offset 计算）。

#### Phase 4d 当前可观察事实

| 配置 | 撑过 launches | 结局 |
|---|---|---|
| 旧（captured memcpy + AUTO_FREE + 单次 upload） | 1 | 第 2 次 hang |
| Slot fix（每 slot 独立 host 区间）+ 单次 upload + AUTO_FREE | 1 | 第 2 次 ILLEGAL_ADDRESS（sync 时） |
| Pre-populate（无 captured memcpy）+ 单次 upload + AUTO_FREE | 1 | 第 2 次 hang |
| Pre-populate + per-launch upload + AUTO_FREE | **14** | 第 15 次 SIGSEGV |
| Pre-populate + per-launch upload + flags=0 | 4 | 第 5 次 SIGSEGV |

→ 即使 graph capture 全跑通，理论收益仅 10-15% TPOT（一次替换掉部分 host
overhead），与 vLLM 差距是 2.2× ~ 3.6×，性价比不够。

### Phase 5 — c=4 plateau 调查（commit 1afdd2c + da2f725）

加 profiling 钩子（FERRUM_BATCH_DECODE_PROF=1），跑 c=4 bench，pod 输出：

```
[batch-decode-prof] m=4 total=10.7ms prep=1us decode=10.2ms post=0.42ms
                    (logits=67us lock=0us sample=326us emit=19us)
[iter-prof]         iter#N total=10.7ms sched=4us process=10.7ms batch_size=4
```

| Phase | 时间 | 占比 |
|---|---|---|
| `scheduler.next_batch` | 4 μs | 0% |
| `process_batch`（= run_batch_decode） | **10.7 ms** | 96% |
| ↳ `executor.batch_decode`（forward + dtoh） | 10.2 ms | 95% of run_batch_decode |
| ↳ post（logits 67μs + sample 326μs + emit 19μs） | 0.42 ms | 4% |
| iter total（engine 内部） | 10.7 ms | |
| TPOT_p50（client 测量） | 13.0 ms | |
| **gap（HTTP/SSE/tokio 调度）** | **2.3 ms** | 18% of TPOT |

→ engine 内部 OK，瓶颈是 decode forward 本身（matmul 主导）。
