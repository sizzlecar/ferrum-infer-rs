# Retrospective — Chunked-Prefill + CUDA Graph Capture (Phase 13)

**Date:** 2026-05-05
**Branch:** `bench/v0.2-cuda`
**Net result:** ferrum 510 → 714 tok/s c=16 (+39%), gap to vLLM closed from 26% → 36%.

This is a self-review of the assistant's mistakes during the session, written so the next session doesn't re-make them.

## What went right

- **Atomic commits with clear messages.** Each bug got its own commit with the reasoning in the body. `git log` is readable as a story.
- **Profile-driven targeting.** After Step 3b minimal landed, the per-op profile showed matmul 65% / attn 20% / split-fused 8% — guided the next move (varlen single-launch on split-fused, then graph capture for launch-overhead reduction).
- **Apples-to-apples vLLM bench.** Re-measured vLLM on the same pod with the same client invocation instead of trusting the stale "1597 tok/s" baseline from earlier docs. Real number was 1986 → ratio went from "we're at 41%" to "we're at 33%" honest. Hard ground truth matters.
- **Pre-warm pattern for process-global pools.** Once the graph-vs-alloc bug was understood, the fix (`pregrow_marlin_gather_scratch`) is reusable for any future process-global slot that grows on demand.

## What went wrong (in chronological order)

### 1. `size_of::<f32>()` on f16 buffers (`cfc0170`)

The per-item byte_offsets in unified_forward_layer were `q_dim * size_of::<f32>()` = 4 bytes/elem. Buffers are `CudaSlice<f16>` = 2 bytes/elem. Offsets were 2× too large. For tokens=1 the i-th item's offset still landed inside the right q_dim row (because the stride matched i's intended slot at i × q_dim under either calculus when bytes_per_elem cancels), but for tokens>1 chunked prefill it skipped every other token row.

**Why I missed it:** copied the byte_offset pattern from Phase 4b's `forward_layer_batched_decode_post_attn` without re-deriving it. Phase 4b ALWAYS calls with tokens=1 so the bug never showed up there.

**Lesson:** when copying a pattern from a code path with a known-narrow input shape (tokens=1), re-derive it for the new path's shape (tokens=q_lens[i] up to ~256).

### 2. Q output layout: head-major vs token-major (`cfc0170`)

My new fused kernel wrote Q in head-major `[q_heads, tokens, hd]`. `paged_varlen_attention.cu` reads Q in token-major `[total_q, num_heads, hd]`. For tokens=1 the layouts collapse; for tokens>1 they read interleaved garbage.

**Why I missed it:** the variable name `q_head_major` in the unified scratch was misleading — I assumed the downstream kernel matched. Should have read `paged_varlen_attention.cu`'s docstring before writing the fused-write kernel.

**Lesson:** when writing a producer kernel for an existing consumer kernel, READ the consumer's layout doc first. Variable names lie; kernel signatures don't.

### 3. Index buffers sized to first call's batch (`cc0f976`)

`ensure_unified_scratch` allocated `unified_cu_seqlens_q`, `unified_pos_offsets`, `unified_block_tables`, `unified_packed_*` to `num_seqs+1`/`num_seqs`/etc. on the FIRST call. Server warmup is c=1 → buffers sized to 2 ints / 1 int / 128 ints. Bench at c=16 then writes 17 ints into a 2-int buffer → `cuMemcpyHtoD failed: CUDA_ERROR_INVALID_VALUE`.

**Why I missed it:** the `ensure_unified_scratch` function had a fast-path `if m_total <= self.unified_capacity { return; }` that gated regrow on m_total but not on num_seqs. I'd thought of one dimension.

**Lesson:** when scratch buffers are sized along multiple axes (m_total AND num_seqs), gate the early-return on ALL of them. Better still, size to the engine-known cap once at startup so first-call sizing isn't load-bearing.

### 4. Pool exhaust → silent ensure_kv → panic (`3cf3bf6`)

`ensure_kv` `eprintln!`d on pool-exhaust and returned silently. `unified_forward_internal` then asserted `kv_caches[cid]` exists and panicked. The panic killed a tokio worker, dangling every cache_id still in flight on that worker → cascading "paged KV pool exhausted" spam → bench wedged.

**Why I missed it:** treated the silent-return + assertion-panic pair as fine because each call site was "correct in isolation." Didn't think about what happens when pool ACTUALLY exhausts.

**Lesson:** if a function silently fails (no Result, no panic, just return), the next caller MUST defensively check. Better: don't silent-fail in the first place — return Result and propagate.

### 5. THE cache_id leak (`1645663`)

Engine's `run_batch_decode` set `UnifiedBatchItem.seq_id = rid.to_string()` (the engine's request_id, a UUID). Executor's `unified_decode` uses that as `cache_id` when calling `model.unified_forward(...)`. But the model's `kv_caches` is keyed by `seq.model_cache_id` = the executor-generated `llm-cache-N` (set during prefill). Mismatch → `ensure_kv(<UUID>)` allocated a brand-new cache + 128 paged blocks every decode iter. UUID-keyed cache had a fresh-paged-blocks set on every iter and never released (no completion path knew about the UUID-keyed entry).

Visible as: bench rep 1 succeeded (650 tok/s), rep 2 collapsed to 211 tok/s, rep 3 to 10 tok/s. Server log: `[ferrum] paged KV pool exhausted on ensure_kv for cache_id="llm-cache-60"` — counter went past 60 unique caches even though only 32 prompts had run.

**Why I missed it:** there are TWO id namespaces — engine `request_id` (UUID) and executor `cache_id` (`llm-cache-N`). They share a `String` type. I picked one without checking which the model was actually keyed on. The function name `seq_id` didn't clarify.

**Lesson:** when two layers each generate identifiers and they meet at a trait boundary, the boundary doc MUST say which one is the canonical key. Newtype the two id types so the compiler refuses to mix them. Until the type system enforces it, write a debug_assert that the cache_id passed to the model matches what the model issued.

### 6. CUDA Graph: missing post-capture replay (`f1f0f0b`)

Capture only RECORDS launches into the graph; the kernels do NOT execute. My first capture cut went `begin_capture → layer_loop → end_capture` and assumed the eager pass had executed alongside. It hadn't — `packed_logits` stayed uninitialised, and the next iter's gather_columns (eagerly run on stale stream state) failed with `CUDA_ERROR_INVALID_VALUE`.

**Why I missed it:** the legacy `forward_layer_batched_decode_post_attn` does `end_graph_capture` immediately followed by `replay_graph` for the FIRST iter. I copied the structure but dropped the immediate-replay line.

**Lesson:** post-capture replay isn't optional. The pattern is `begin → record → end → IMMEDIATE REPLAY → next-iter REPLAY again`. The immediate replay is what actually runs the recorded work for the iter that captured.

### 7. CUDA Graph: alloc-during-capture (`1cde3af`)

`with_marlin_gather_scratch` does an in-place `stream.alloc::<f16>` when the slot is too small. cuLaunchKernel after a runtime alloc inside a captured stream returns `CUDA_ERROR_INVALID_VALUE` — capture rejects runtime resource ops. Sequence: warmup at m≈4 sized the slot to 14k elems; first captured iter at m=256 prefill needed 3.67M → grew DURING capture → broken.

**Fix:** new `Backend::pregrow_marlin_gather_scratch(ctx, m_total * intermediate_size)` called eagerly BEFORE `B::sync()` and `begin_graph_capture`.

**Why I missed it:** the existing process-global slot was a fix from a PRIOR session for a different bug (cudarc pool growth). I knew it allocated lazily but didn't connect "lazy alloc" to "alloc inside capture is illegal."

**Lesson:** ANY process-global resource pool that allocates lazily is a hazard for graph capture. Audit them all at the start of any graph-capture work and add explicit pregrow APIs.

### 8. Wasted GPU time on `/loop` self-pacing

Early in the session I let `/loop` schedule wakeups every 60-120s even when there was nothing actionable yet. The pod was billed by the hour and idle GPU minutes accumulated to ~4 hours of wasted spend before the user pulled me up on it.

**Lesson:** when you're waiting for a build/bench, ARM A MONITOR (event-driven) and use ScheduleWakeup as a fallback heartbeat at 1200-1800s — never as the primary signal. The `/loop` skill doc actually says this; I ignored it.

### 9. vllm-marlin re-test was wasted

The Phase 12 doc said vllm-marlin gave "1% throughput diff" on this exact workload. I rebuilt the binary with `--features cuda,vllm-marlin` (+18 minutes) and re-benched anyway, hoping the chunked-prefill changes had altered the picture. They hadn't — same 1% diff. 18 minutes that should have gone into either CUDA Graph or the attention kernel.

**Lesson:** if a prior session has a clear conclusion ("X doesn't help on this workload"), don't re-test under cosmetic changes. Re-test only if the underlying bottleneck has shifted (and the profile data should justify the bet).

### 10. Heredoc-via-SSH dropped a script

Used `ssh ... 'cat > /tmp/script.sh <<EOF ... EOF'` to write a profile script — the heredoc didn't survive SSH's command-line parsing in some attempts, the file was empty, the script ran "No such file." Cost a wakeup cycle to rediscover.

**Lesson:** when writing files over SSH, pipe the heredoc via stdin (`ssh ... 'cat > /tmp/script.sh' <<'EOF' ... EOF`), not as part of the remote argv.

### 11. SSH connection flakiness wasn't handled

Multiple `ssh` calls returned exit 255 mid-session (network blip on Vast.ai). Each made me re-run, sometimes leaving stale state on the pod. Should have wrapped critical state-setup commands with retry logic from the start.

**Lesson:** treat the SSH layer as flaky. Idempotent commands on the pod side, retries on the local side.

## Patterns to keep

- **Smoke before bench.** A single-prompt curl request validates the path end-to-end in 5 seconds. Multiple times this session, a smoke would have caught a bug 3-5 minutes faster than the bench did.
- **Per-op profile counters as part of any new path.** The drain-every-N pattern (`fd75525`) is cheap to add and made the matmul-vs-attn-vs-split breakdown obvious. Add this to ANY new model-forward variant.
- **Document failure modes inline.** The status comments at the gate-check sites (`5f5166d`, `8b28adf`) act as breadcrumbs for the next session — they explain WHY a flag exists and what tripped it last time.
