# M3 A/B Runner Status - 2026-05-30

Milestone C is not complete. This checkpoint creates the reusable runner
surface and migrates M3 wrappers away from copied server/gate/cleanup logic.

## Added

- `scripts/m3_ab_runner.py` centralizes:
  - optional release build;
  - preflight capture (`git`, binary SHA256, GPU/process state);
  - server launch and health wait;
  - Paris and optional multi-turn gates;
  - `bench-serve`;
  - per-case `manifest.json`;
  - root `manifest.json`;
  - cleanup status;
  - explicit `artifact_verdict`, `not_publishable`, and
    `not_publishable_reason` metadata;
  - stable sorted runtime config snapshot derived from preset/base/case
    `FERRUM_*` env, with runner-injected values marked `source=script_case`;
  - log-derived `profile.jsonl` for profile wrappers as a transition artifact;
  - structured profile JSONL mode backed by `FERRUM_PROFILE_JSONL`;
  - case-scoped profile sink metadata injection via
    `profile.profile_env_cases`;
  - summary JSON and console table.
  - machine-readable `runtime_config_diff_vs_baseline` in `summary.json`,
    including env hashes plus added/removed/changed runtime config entries.
  - machine-readable `performance_regression_gates` in `summary.json` when a
    `baseline_case` is configured. The gate records throughput and latency
    deltas, per-metric thresholds, pass/fail status, and the per-case aggregate
  result.
  - runner-owned process leak detection after cleanup. The scanner treats
    leftover server descendants, case-matched `bench-serve` clients, and
    case-port server processes as cleanup failures.
  - global post-cleanup process hygiene recording for residual
    `ferrum serve`, `bench-serve`, `cargo`, `nvcc`, and `vllm` processes.
    Publishable artifacts fail validation when this scan is not clean.
  - per-case `effective_config.json` and `decision_trace.jsonl` artifacts.
    The current decision trace is a runner-level selector skeleton for the
    M3 fast-path controls: prefill/mixed attention, decode attention, MoE
    implementation, MoE graph policy, KV block count, max sequences,
    max batched tokens, prefix cache policy, scheduler admission, and sampling
    readback.
  - server-side config artifact emission. Runtime cases now pass
    `--effective-config-json` and `--decision-trace-jsonl` to `ferrum serve`
    and record the decision count from the server-written JSONL trace.
  - per-run and per-case `validation_checklist` records for Milestone I:
    change type, touched areas, required correctness gates, observed gates,
    bench completion gate, local/skipped gates, baseline case, and whether a
    performance-regression gate is required.
- `scripts/m3_validate_runner_artifact.py` validates runner artifact roots:
  root manifest, per-case manifests, summary rows, runtime config snapshots,
  correctness gates, benchmark completion/error metrics, cleanup status, and
  optional profile JSONL events. It also validates artifact verdict and
  publishability metadata. Cleanup validation now rejects runner-owned process
  leaks. It now also rejects missing/mismatched effective-config and decision
  trace artifacts, including missing required selector decisions. Decision
  trace validation now rejects unknown sources, non-`FERRUM_*` source keys,
  empty candidates/affects, and malformed rejected-candidate reasons. Profile
  JSONL validation now mirrors the `ferrum-bench-core::profile::ProfileEvent`
  envelope checks. Effective-config validation now checks the startup artifact
  schema, `sha256:` env hash, sorted `FERRUM_*` entries, entry
  `source`/`affects` values, and decision parity with `decision_trace.jsonl`.
  Runtime snapshot and diff validation now also require `FERRUM_*` keys,
  allowed source/effect values, stable ordering, and non-empty `affects` when
  present.
  Cleanup validation now also rejects global post-run process hygiene
  findings, not only runner-owned leaks.
  Publishable artifacts now also fail validation if the checklist skips a
  required gate, omits a required correctness gate, records bench errors, or
  requires performance regression evidence without an enabled baseline gate.
  Validation checklists now use a controlled `touched_areas` vocabulary and
  the validator derives minimum required gates from those areas. For example,
  model-forward, attention-path, FA2-runtime, scheduler, runtime-default, and
  CUDA-kernel artifacts must declare at least `paris` plus
  `bench_completion`; profile/benchmark-harness artifacts must declare
  `bench_completion`.
- The runner includes the named runtime preset `m3_qwen3_30b_a3b_int4`.
  Wrappers now set the preset plus only case-specific overrides, with
  diagnostic/profile wrappers adding their diagnostic knobs separately.
- Structured profile sink metadata is passed to `ferrum serve` through typed
  `--profile-*` CLI args instead of runner-injected `FERRUM_PROFILE_*` env
  variables. Per-case manifests still record `profile_env_enabled` for
  compatibility with existing artifacts.
- Runner-side runtime config snapshots now filter out non-runtime variables
  such as `HF_HOME` and record script-injected preset/base/case `FERRUM_*`
  values as `source=script_case`.
- Migrated wrappers:
  - `scripts/m3_fa2_direct_ffi_ab.sh`
  - `scripts/m3_fa_layout_varlen_ab.sh`
  - `scripts/m3_attn_v1_ab.sh`
  - `scripts/m3_route_unified_profile.sh`
  - `scripts/m3_graph_runtime_profile.sh`
  - `scripts/m3_prefill_first_ab.sh`
  - `scripts/m3_active_prefill_chunk_ab.sh`
  - `scripts/m3_profile_sink_overhead_ab.sh`
- Wrappers support `VALIDATE_ONLY=1`, which validates generated runner
  config without launching a server.
- CPU CI now runs the runner/artifact-validator self-tests plus
  `VALIDATE_ONLY=1` checks for all migrated wrappers, so broken runner config
  schema, validation checklist handling, or wrapper JSON generation fails
  before any GPU artifact is attempted.
- `scripts/m3_route_unified_profile.sh` now uses structured profile mode with
  required event groups (`moe_dump`, `vllm_moe_config`, `unified_prof`,
  `iter_prof`, `bucket_prof`, plus `unified_layer_prof` or
  `batched_decode_prof`) and marks its graph-off sync-timer output
  `diagnostic-only`.
- `scripts/m3_graph_runtime_profile.sh` now uses the same runner in structured
  profile mode. It keeps the production graph path enabled, validates
  `unified_prof`, `iter_prof`, `graph_prof`, and `batched_decode_prof` or
  `unified_layer_prof` events from `profile.jsonl`, and marks the run
  `diagnostic-only`.
